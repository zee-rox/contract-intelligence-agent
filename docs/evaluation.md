# Evaluation

The repository includes a reproducible evaluation harness under `backend/eval`.

## Goals

The harness measures the behaviors that distinguish the project from a basic chat-with-PDF demo:

- Clause extraction precision, recall, and F1.
- Clause boundary support against source snippets.
- Clause-level risk accuracy.
- Citation validity.
- Refusal accuracy for unsupported questions.
- OCR evaluation status.
- Provider comparison metadata.

## Run The Harness

```bash
cd backend
python -m eval.runner
```

Outputs:

- `EVAL.md`: human-readable report.
- `backend/eval/results/latest.json`: machine-readable result payload.

## Dataset

The current dataset is `phase5-synthetic-v1`. It uses small synthetic contracts and annotations so the project can be evaluated without confidential or copied contract text.

The dataset covers:

- Expected clause categories and source snippets.
- Risk expectations.
- Supported and unsupported questions.
- Citation validation against normalized source text.

It does not yet include a scanned OCR fixture, so OCR degradation is reported as not evaluated.

## Provider Comparison

The fake provider runs by default for deterministic local evaluation.

To include a llama.cpp-compatible provider comparison:

```bash
export EVAL_RUN_LLAMA_CPP=true
export LLAMACPP_BASE_URL=http://localhost:8080/v1
cd backend
python -m eval.runner
```

The harness records skipped provider runs explicitly rather than inventing metrics.

## Known Evaluation Gaps

- The dataset is intentionally small.
- OCR behavior is not scored until a scanned fixture is added.
- The current refusal score highlights one unsupported-answer issue in `phase5-synthetic-v1`.
- Real-provider outputs can vary; deterministic provider results are the baseline for repository tests.
